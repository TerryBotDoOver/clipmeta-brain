## Bugs found and fixed tonight

### 1. Windows cp1252 Unicode crashes — a full family
Same underlying cause as the `❌` bug fixed 4/12 — Python's default stdout encoding on Windows is cp1252 which can't encode characters like `→ ✅ ❌`. Third print statement of this family crashed the first real run mid-Stage-5.

**Fixed:**
- `run.py` top-level: `sys.stdout.reconfigure(encoding="utf-8")` and same for stderr — bulletproofs ALL future unicode in print statements
- `render.py` — replaced `→` with `to` in two log statements (lines 209 and 306)
- `run.py` — replaced `→` with `->` in mux log
- `validate.py` — replaced `✅`/`❌` with `[OK]`/`[FAIL]`
- `api/main.py` `trigger_run` — now sets `PYTHONIOENCODING=utf-8` and `PYTHONUTF8=1` on the subprocess env
- `api/main.py` `trigger_run` — captures stdout/stderr to timestamped log file at `logs/{slug}_{ts}.log` instead of `DEVNULL` so crashes are debuggable

### 2. Predator SSH mkdir bug
Pipeline's `render_remote_predator` was using `mkdir -p` against the Predator's default PowerShell shell. PowerShell's `mkdir` does NOT support `-p`. Command exited code 1 silently and the pipeline fell back to Dell libx264 — which made the first real run take ~19 hours instead of ~2 hours.

**Fixed:** replaced `mkdir -p path1 path2 path3` with PowerShell-native:
```
New-Item -ItemType Directory -Force -Path 'path1','path2','path3' | Out-Null
```
Tested end-to-end via SSH from Dell → Predator before restarting the pipeline. Works.

### 3. Predator SSH silent fallback
When the Predator path failed, the pipeline logged a soft one-liner and silently fell back to Dell libx264. Took ~35 minutes of watching the wrong GPU do nothing to figure this out.

**Fixed:** the fallback now logs a loud multi-line warning with the error type, exit code, stdout, and stderr of the failed SSH command. Can't miss it in the log.

### 4. Wrong Predator LAN IP in config
Config had `ssh_host: levib@10.0.0.158` but the actual Predator LAN IP is `10.0.0.157`. The Tailscale hostname `ef-productions` resolves to `100.110.229.98` (tailnet) with direct LAN route to `10.0.0.157`.

**Fixed:** updated `still_frequency_config.json` to `10.0.0.157`.

### 5. Predator SSH key auth missing
Dell's `~/.ssh/id_ed25519.pub` was not in the Predator's authorized_keys. ALSO — critical Windows OpenSSH gotcha — Administrators authenticate via `C:\ProgramData\ssh\administrators_authorized_keys`, NOT the user's `.ssh\authorized_keys`. Levi initially added the key to the wrong file.

**Fixed:** Levi added the Dell's ed25519 pub key to the Predator's `administrators_authorized_keys` file with correct NTFS permissions (via `icacls` with Administrators + SYSTEM only). Verified hostname `EF_Productions`, ffmpeg 8.0.1 path, and NVENC h264 encoder all reachable via SSH.

### 6. Shuffle randomness flakiness
Shuffle was rolling a random seed per invocation. Dry-run used seed 42 (deterministic preview) and passed; first real run rolled a different seed and hit 1 violation at the hard rule that couldn't be repaired. Pipeline immediately failed.

**Fixed:** patched `run.py` to try 6 known seeds (`[42, 1337, 7, 99, 2024, 8128]`) in order before declaring shuffle infeasible. First seed used is always 42, matching the dashboard preview, so dry-run and real-run are now consistent.

### 7. Resolution config was 1080p, not 4K
`still_frequency_config.json` had `resolution: 1920x1080`. Levi caught this mid-session — he wants 4K final output because the drone footage is 4K source and 4K gets better YouTube RPM.

**Fixed:** config changed to `3840x2160`, NVENC CQ bumped from 21 → 23 (4K at CQ 21 would be ~100GB+ for a 10h file; CQ 23 drops size by ~30% and still looks excellent at 4K).

### Zoho API patterns learned (saved to [[howto|howto.md]])
- `PUT /expenses/{id}` — update account on already-categorized bank feed items
- `POST /banktransactions/uncategorized/{id}/categorize` — works for `transfer_fund`, `sales_without_invoices`, `expense_refund`, `owner_drawings`. **Does NOT accept `expense` as a transaction type.**
- For regular expenses: `POST /expenses` to create the record, then `POST /banktransactions/uncategorized/{bank_id}/match` with the new expense_id
- To unwind a bad categorization: `POST /banktransactions/{imported_transaction_id}/uncategorize` — uses the feed item's ID, not the transaction wrapper's ID. Returns the feed to uncategorized + deletes the attached transfer/expense record.
- **Critical rule for transfers between two fed accounts:** categorize ONE side only. The other side must be MATCHED to the created transfer record via the match endpoint, not independently categorized, or you get duplicate double-entry.
