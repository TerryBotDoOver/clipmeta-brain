## Bugs Fixed

### Upload limit triggered too late
**Problem:** Free users could upload an entire video file to R2, then get blocked at the clip record creation step. Wasted bandwidth and bad UX.
**Root cause:** Limit was only checked in `POST /api/clips` (after R2 upload). Also, the upload-url fetch happened AFTER frame extraction, which loaded the full video into the browser.
**Fix:**
- Added clip count limit check to `POST /api/clips/upload-url` (server-side)
- Reordered `UploadForm.tsx` so the upload-url call runs FIRST per file, before codec detection or frame extraction
- 429 response now blocks instantly with no video loading

### Regen vs upload limit collision
**Problem:** When regenerating a clip on free plan, users were getting "Daily clip limit reached" instead of the regen limit message. Even regenerating a single clip after 3 uploads was being blocked.
**Root cause:** The clip upload limit check ran for ALL generation requests, including regenerations. The check counted all clips in the table from today, which always included the user's existing clips.
**Fix:** Removed clip upload limit check from `generate/route.ts` and `generate-worker/route.ts` entirely. Upload limits are now enforced ONLY at clip creation time. The generate routes only enforce regen limits.

### Regens_used double-counted as regenerations
**Problem:** First-time generations were being counted as regenerations. User showed "1/1 regenerations used" after just uploading clips, never regenerating.
**Root cause:** The old formula `regensUsed = clips_used_this_month - clip_history_count` was fundamentally broken because nothing inserts `clip_history` rows with `action='created'`. Plus `clips_used_this_month` was being incremented at both upload AND generation time.
**Fix:** Created dedicated `regens_used_this_month` counter on profiles. Only increments when `isRegeneration === true`. Removed all subtraction-based regen calculation. Reset alongside `clips_used_this_month` in billing webhook.
**SQL migration applied:** `alter table profiles add column if not exists regens_used_this_month integer not null default 0;`

### Dashboard showed deleted projects
**Problem:** Soft-deleted projects appeared in "Recent Projects" list and inflated the project count.
**Root cause:** Dashboard SQL query didn't filter `.is("deleted_at", null)` like the projects page did.
**Fix:** Added the filter to the dashboard projects query.

### Existing-account signup UX
**Problem:** When user tried to sign up with an email that already existed, Supabase returned success and sent a "magic link" email, and the app showed "Check your email for confirmation" -- confusing.
**Root cause:** Supabase's `signUp` returns success with `identities: []` (empty array) when email exists, to prevent email enumeration. The code didn't check this.
**Fix:** Detect empty identities array, show "An account with this email already exists" with inline Sign In and Reset Password links.

### Upload progress bar disappeared
**Problem:** When uploading, status went straight from "extracting" to "AI" to "done" with no progress bar visible during the actual upload.
**Root cause:** Status was set to "extracting" during frame extraction, but never set back to "uploading" before the actual XHR upload started. The progress bar was only rendered when `status === "uploading"`.
**Fix:** Added `updateFile(item.id, { status: "uploading", progress: 0 })` right before the actual upload step. Also rebuilt the progress bar with a glowing purple/pink/amber gradient and animated shimmer overlay sliding across the filled portion.

### Referral tracking was completely broken
**Problem:** Pending referral count never updated when a new user signed up via referral link.
**Root cause:** TWO bugs.
1. The `?ref=` query param was overloaded: auth page checked if it equals "pricing" (UTM source), and ignored everything else. The referral code was never captured.
2. Nothing in the entire codebase actually called `/api/referral/track`. The endpoint existed but was never invoked from the frontend.
**Fix:**
- Auth page now detects 8-hex-char `ref` param, saves to localStorage as `pendingReferral`
- New `ReferralTracker.tsx` client component mounted on dashboard reads localStorage and POSTs to `/api/referral/track`
- Updated track route to only set `referred_by` if currently NULL (don't overwrite)

### Referral qualify cron crashing every run
**Problem:** No referral has ever been processed in the history of the product.
**Root cause:** The cron's main SELECT referenced `email` and `created_at` columns on the `profiles` table, but neither column exists. Email lives in `auth.users`. The query crashed every time the cron ran.
**Fix:** Removed `email` and `created_at` from the profiles select. Now fetches email separately via `supabaseAdmin.auth.admin.getUserById()` for each candidate. The disposable email anti-fraud check still works.

## Features Built

### Welcome Promo Reward (50% off, 24h countdown)
**What it does:** When a new user completes the onboarding checklist, they unlock 50% off their first month for 24 hours. The discount is enforced server-side.

**Components:**
- Migration: `promo_unlocked_at timestamptz` on profiles
- `/api/onboarding/complete` sets `promo_unlocked_at` only on first completion
- `/api/billing/checkout` checks the 24h window and attaches Stripe coupon `welcome_reward_50` if eligible
- New `PromoReward.tsx` client component shows fireworks popup + pill modal + 24h countdown timer
- After popup is closed, top banner shows with countdown and "Claim Now" button
- After 24h, banner disappears and server stops applying the coupon
- Listens for custom event `clipmeta:onboarding-complete` to re-fetch profile mid-session
- Old `UpgradeBanner.tsx` (FOUNDING50) disabled

**Stripe coupon created:** `welcome_reward_50` (50% off, once duration). Created via Stripe API using restricted key with Coupons:Write scope.

**Onboarding checklist visual update:** Added gift icon at end of progress bar that bounces and glows when all 4 steps are complete. Subtitle updated to mention the 50% off reward.

### LimitReachedModal (shared across upload + regen)
**What it does:** Replaces all `confirm()` browser dialogs for limit-reached errors with a centered pill modal in the same gradient style as the promo reward.

**Wired into:**
- `UploadForm.tsx` (clip upload limit)
- `GenerateMetadataButton.tsx` (single regen limit)
- `ReviewQueue.tsx` (bulk regen limit)
- `BulkGenerateButton.tsx` (still uses inline error, kept as-is)

Modal has lightning bolt icon, title, message, optional value-framed upgrade message in darker box, "Upgrade Plan" button (white pill) and "Maybe Later" button. Click backdrop or X to close.

### Bonus clips celebration animation
**What it does:** When a referrer's `bonus_clips` increases between dashboard loads, fires a fireworks popup celebrating the new clips earned.

**How it works:** `ReferralCard.tsx` saves the last-seen `clipsEarned` value to localStorage. On next load, compares to current value. If higher, shows centered pill modal with "+X bonus clips" and 6-second sparkle animation.
